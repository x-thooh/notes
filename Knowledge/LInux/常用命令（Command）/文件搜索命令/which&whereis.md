## 区别

- **`which`**：只看**当前用户的 `$PATH`**，只找**可执行文件**，返回第一个匹配的路径。
    
- **`whereis`**：不看 `$PATH`，而是去**系统预定义的标准目录列表**（如 `/bin`、`/usr/bin`、`/usr/local/bin`、手册页目录、源码目录）里找，返回**所有**匹配的**二进制、源码、手册页**。
    

---

## 对比

|对比项|`which`|`whereis`|
|---|---|---|
| **搜索依据**|当前 Shell 的 `$PATH` 环境变量|硬编码/预定义的系统标准路径（加上 `$MANPATH` 等）|
|**搜索对象**|仅**可执行文件（二进制）**|**二进制** + **源码** + **手册页**|
|**返回结果**|默认只返回**第一个**匹配（可用 `-a` 显示全部）|返回**所有**找到的路径|
|**是否受 `$PATH` 影响**|**完全受**影响|**基本不受**影响（只看系统固定目录）|
|**典型场景**|检查我敲的这个命令，到底执行的是哪里的文件|查一个命令相关的所有文件都散落在哪|

---

## ⚠️ 两个至关重要的“坑”

### 坑 1：`which` 不一定靠谱

很多系统上，`which` 本身是一个**外部程序**（`/usr/bin/which`），它启动时会读取你的 `$PATH`。但它**不识别 Shell 内置命令（builtin）和别名（alias）**。

比如你设置了 `alias ll='ls -al'`，敲 `which ll` 大概率会报错“找不到”，但实际上 `ll` 是好用的。  
✅ **更推荐用**：`type -a` 或 `command -v`，它们是 Shell 内置的，能显示别名、内置函数和外部命令的全部情况。

```
type -a gcc   # 显示所有匹配（别名、内置、外部）
```

### 坑 2：`whereis` 可能找不到你手动装的程序

如果你自己编译软件装到了 `/opt/myapp/bin`，并且把它加到了 `$PATH` 里，**`which` 能找到**，但 **`whereis` 很可能找不到**，因为 `/opt/myapp/bin` 不在它的“系统标准目录”白名单里。

## 常用实用参数

|命令|参数|作用|
|---|---|---|
|`which`|`-a`|显示 `$PATH` 中**所有**匹配的路径（默认只显示第一个）|
|`whereis`|`-b`|**只找**二进制文件（不显示源码和手册）|
|`whereis`|`-m`|**只找**手册页（man）|
|`whereis`|`-s`|**只找**源码|
